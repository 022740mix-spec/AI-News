# 執筆ガイド — 取材から公開までの全ルール

## 1. ソースの優先順位と取材

### 信頼度順

1. 公式プレスリリース・ブログ（企業サイト、GitHub リリースノート）
2. 主要通信社・新聞（Reuters, AP, NYT, Bloomberg）
3. 技術メディア（The Verge, TechCrunch, Ars Technica）
4. 開発者コミュニティ（Hacker News, X/Twitter の公式アカウント）

### 掲載判断

- AI 開発ツール・モデル・企業に関連するか
- 読者（日本語圏の開発者）にとって実用的か
- 事実として確認可能か（「噂」「リーク」は明示する）

### 取材カテゴリ

| カテゴリ | 具体例 |
|---------|--------|
| バイブコーディング | Claude Code, Cursor, Windsurf, Copilot, Codex の新機能 |
| AI モデル | GPT, Claude, Gemini, Llama の新バージョン・ベンチマーク |
| 画像/動画/音楽生成 | Midjourney, DALL-E, Runway, Sora, Suno, Udio |
| 音声入力・TTS | Aqua Voice, Whisper, ElevenLabs |
| フィジカル AI | ロボティクス, 自動運転, エッジ AI |
| 社会・規制 | AI 規制, 訴訟, 企業動向, 資金調達 |
| 実用スキル | スキルファイルの書き方, CLAUDE.md, RAG, 音声コマンド活用 |

## 2. ファクトの構造化

収集した事実を以下の形で整理する:

| 項目 | 記入内容 |
|------|---------|
| 何が起きたか | 主語 + 動詞 + 目的語（一文） |
| いつ | 日付（YYYY-MM-DD、東京時間基準） |
| 誰が | 企業名・人名（`aiCompanies.js` の `id` と紐付け確認） |
| ソース | URL + サイト名（`primarySources` 形式で記録） |
| 関連する既存記事 | `?a=article-id` があれば記録 |

## 3. 記事データフォーマット

```javascript
{
  id: "kebab-case-slug",          // 一意のスラッグ
  type: "news",                   // news | feature | review
  category: "model",              // special | model | cli | editor | data | product | voice | vision | video
  title: "見出し — 簡潔に事実を",
  excerpt: "概要。事実のみ、感想禁止。",
  body: [
    "第1段落 — リード。最も重要な事実。",
    "第2段落 — 補足情報・背景。",
    "第3段落 — 関連リンク ?a=... への導線。"
  ],
  date: "YYYY-MM-DD",
  author: "AI News 編集部",
  readTime: "N分",                // 日本語 400-600文字/分 で算出
  tags: ["関連タグ"],
  primarySources: [
    { title: "ソースタイトル", site: "サイト名", url: "https://..." }
  ]
}
```

## 4. テキスト装飾ルール

| 装飾 | 記法 | 使う場面 | 禁止 |
|------|------|---------|------|
| 太字 | `**text**` | 固有名詞の初出のみ | 1段落3箇所以上は多すぎ |
| ハイライト | `==text==` | 読者に注目してほしい箇所 | 毎段落に入れない |
| 警告 | `!!text!!` | 絶対やってはいけないこと、危険操作 | 通常の注意事項に使わない |
| 通常テキスト | そのまま | 本文の大部分 | -- |

本文がほぼ全部太字だと何も目立たない。大部分は通常テキストで書く。

## 5. ファクトチェック・リンク検証・表記統一

### ファクトチェック

- `primarySources` の URL を開き、公式発表日と記事の `date` が一致するか
- 企業名・製品名が `aiCompanies.js` の `name` / `products` と一致するか
- 数値・バージョン・人名が公式ソースと合致するか
- 引用の内容が元ソースと合致するか

### チャート・グラフデータの検証

- 各 `bars[].value` が `primarySources` に記載されているか
- ==ソースがない数値でチャートを作らない。推測で値を入れるのは禁止==
- !!推定値でグラフを作らない。これは原則禁止!!
- やむを得ず推定値を含める場合（レビュー・特集のみ）は注記に「AI News 編集部による推定を含み、公式発表値ではありません」と明記

### リンク検証

- `primarySources[].url` が全てアクセス可能か
- `body` 内の `?a=article-id` が ARTICLES 配列に存在するか

### 表記揺れ統一ルール

| 項目 | 正 | 誤（よくある揺れ） |
|------|----|-------------------|
| 企業名 | `aiCompanies.js` の `name` | 略称・旧名の混在 |
| 製品名 | 公式表記（大文字小文字含む） | Claude code → Claude Code |
| 日付形式 | YYYY-MM-DD（データ）/ M月D日（本文） | 2026/03/28, 3/28 |
| 敬称 | 原則なし（報道文体） | 統一されていれば可 |
| 句読点 | 「、」「。」 | 「，」「．」混在 |

## 6. 構成・見出し・概要の推敲

### 構成の評価基準

| 評価軸 | 良い状態 | 悪い状態 |
|--------|---------|---------|
| リード | 最重要の事実が第1段落にある | 背景説明から始まっている |
| 情報の階層 | 重要度順（逆ピラミッド）or 時系列 | 並列の羅列 |
| 段落の粒度 | 1段落 = 1トピック | 1段落に複数の無関係な話題 |
| 接続 | 段落間の関係が明示されている | 「。」で切れて脈絡なし |

### 見出し（title）

- 主語 + 動詞 + 何が変わるか を含む
- 20〜40文字。クリックベイト禁止（「驚きの」「ヤバい」「まさかの」）
- 【】タグは記事タイプを示す: 【週刊まとめ】【レビュー】【速報】

### 概要（excerpt）

- 事実の要約のみ。感想・構造説明・メタ文は禁止
- 主語の太字は初出1回のみ。2回目以降は通常テキスト
- 接続詞で時系列・因果・対比を示す。同じ接続パターンを3回以上繰り返さない

### 接続詞のバリエーション

| 関係 | 使える表現（ローテーションする） |
|------|-------------------------------|
| 時系列 | 翌日、前日の〜を受けて、その数時間後に、週末にかけて |
| 対比・並行 | 一方で、同じ日には、法廷では、製品面では |
| 因果 | これを受けて、〜の結果、〜を背景に |
| 追加 | さらに、加えて、別枠では（連続3回禁止） |

### 段落の再構成

- 同じ日の出来事は1段落に束ねる
- 因果関係がある出来事は隣接させる
- 独立した話題が4つ以上並ぶ場合はテーマ別に塊を作る

## 7. 週刊まとめの執筆ルール

### 執筆フロー

1. 対象の7日間（月曜0時〜日曜23:59 JST）を `weekRoundupPeriod` とタイトルで一致させる
2. 掲載基準・配信時刻などの編集ルールは記事本文に書かない
3. カレンダー順に並べ、同じ日は1パラグラフに統合
4. 各トピックに「なぜ重要か（開発者にとって何が変わるか）」を1文付ける
5. 各日: リード（核）→ 補足 → 内部記事リンク `?a=...` の順

### 禁止表現

- 「以下は日付順に〜をまとめる」「一言で言うと」「静かな出だし」「全体としては」（執筆メタ）
- 「今週、」だけの短文列挙
- AI生成っぽい均質な文体（予測可能な段落パターンの繰り返し）

### 悪い例 → 良い例

悪い: 「今週、A が X をした。B が Y をした。C が Z をした。」

良い: 「A は水曜に X を公開した。翌日の木曜には B が Y を発表し、同じ日に C は Z を掲げたため、モデルとツールの告知が重なった。」

## 8. まとめ記事の構成

### 種類

| 種類 | 束ね方 | 例 |
|------|--------|-----|
| 製品別 | 1つの製品の機能・更新を時系列で整理 | 「Claude Code 機能一覧と進化」 |
| 企業別 | 1社の動向を俯瞰 | 「Anthropic の2026年」 |
| テーマ別 | 複数社にまたがるトピックを横断整理 | 「生成音楽ツールの現在地」 |
| 週刊 | 1週間の出来事を時系列で俯瞰 | `weekRoundupPeriod` を設定 |

### 判断基準

- 同一製品/テーマの速報が3本以上 → まとめ記事を検討
- 既存のまとめ記事がある場合 → 更新（追記）を優先、新規作成ではなく

### 構成テンプレート

1. リード — このまとめが扱う範囲と最も重要な変化
2. 時系列 or テーマ別ブロック — 各速報記事の要点を統合
3. 各ブロック末尾に `?a=速報記事ID` で深掘りリンク
4. 締め — 現時点での全体像（感想ではなく事実の要約）

---

## 9. 禁止事項（統合）

- 意見・感想を書かない（「素晴らしい」「期待される」「注目だ」）
- 未確認情報を断定しない（「と報じられた」「と伝えられた」を使う）
- 他記事のコピペ禁止 — 必ず自分の言葉で再構成
- `heroScope` / `pinned` の設定は執筆者はしない（デスク管轄）
- 事実の追加・削除は構成推敲フェーズではしない（取材に差し戻し）
- 編集ルール・運用説明を記事本文に公開しない
