# 構成編集（Editor） — 記事構成・推敲スキル

## 役割の定義

校閲を通過した原稿の**構成・流れ・見出し・概要**を磨く。事実は変えず、**読者にとっての理解しやすさ**を最大化する。

---

## 編集フロー

### Step 1 — 構成の評価

記事を読み、以下を判断する:

| 評価軸 | 良い状態 | 悪い状態 |
|--------|---------|---------|
| **リード** | 最重要の事実が第1段落にある | 背景説明から始まっている |
| **情報の階層** | 重要度順（逆ピラミッド）or 時系列 | 並列の羅列 |
| **段落の粒度** | 1段落 = 1トピック | 1段落に複数の無関係な話題 |
| **接続** | 段落間の関係が明示されている | 「。」で切れて脈絡なし |

### Step 1.5 — テキスト装飾のルール（重要）

**太字（`**text**`）は最小限にする**。本文がほぼ全部太字だと何も目立たない。

| 装飾 | 記法 | 使う場面 | 使わない場面 |
|------|------|---------|-------------|
| 太字 | `**text**` | 見出し、固有名詞の初出のみ | 本文の強調全般（多用禁止） |
| ハイライト | `==text==` | 読者に注目してほしい箇所（背景色が付く） | 毎段落に入れない |
| 警告 | `!!text!!` | 絶対やってはいけないこと、危険操作 | 通常の注意事項 |
| 通常テキスト | そのまま | 本文の大部分 | — |

**判断基準**: 1段落に太字が3箇所以上あったら多すぎ。ハイライトか通常テキストに置き換える。

### Step 2 — 見出し（title）の推敲

**良い見出しの条件**:
- **主語 + 動詞 + 何が変わるか** が入っている
- 20〜40文字（長すぎず短すぎず）
- 【】タグは記事タイプを示す（【週刊まとめ】【レビュー】【速報】）
- クリックベイト禁止（「驚きの」「ヤバい」「まさかの」）

**見出しパターン**:
```
速報:   「OpenAI、GPT-5.4 mini と nano を同時リリース」
レビュー: 「【レビュー】Cursor 2.4 — 並列サブエージェントの実力」
週刊:   「【週刊まとめ】AI開発ツール — 2026年3月23日〜29日」
特集:   「生成音楽ツール比較 — Suno・Udio・Lyria の現在地」
```

### Step 3 — 概要（excerpt）の推敲

**ルール**（`weekly-news-roundup` スキルと共通）:
- 事実の要約のみ。感想・構造説明・メタ文は禁止
- 主語の太字は初出1回のみ。2回目以降は通常テキスト
- 接続詞で時系列・因果・対比を示す
- 同じ接続パターンを3回以上繰り返さない

**悪い例**:
> 今週は色々な発表がありました。以下にまとめます。

**良い例**:
> **OpenAI** は火曜に GPT-5.4 mini/nano を出し、**翌木曜**には **Astral** 買収を公表した。**同じ金曜**に **Cursor** が Windsurf 買収完了を伝えた。

### Step 4 — 本文の推敲

**接続詞・副詞句のバリエーション管理**:

| 関係 | 使える表現（ローテーションする） |
|------|-------------------------------|
| 時系列 | 翌日、前日の〜を受けて、その数時間後に、週末にかけて、〜の翌週 |
| 対比・並行 | 一方で、同じ日には、法廷では、製品面では、開発者向けには |
| 因果 | これを受けて、〜の結果、〜を背景に、〜が材料となり |
| 追加 | さらに、加えて、別枠では、もう一つ（**連続3回禁止**） |

**段落の再構成ルール**:
- 同じ日の出来事は**1段落に束ねる**（日付見出しがある場合）
- 因果関係がある出来事は**隣接させる**
- 独立した話題が4つ以上並ぶ場合は**テーマ別に塊を作る**

### Step 5 — 図表・補足の配置提案

本文だけでは伝わりにくい場合、以下を提案する:
- `tables` の追加（比較・スペック）
- `figures` の追加（構成図・タイムライン）
- `coverImage` の候補（UI担当に引き継ぐ）

---

## 編集の権限と制限

**できること**:
- 段落の並び替え・統合・分割
- 見出し・概要の書き換え
- 接続詞の追加・変更
- 図表の追加提案
- `type` / `category` の変更提案

**できないこと**:
- 事実の追加・削除 → 記者（Reporter）に差し戻し
- 公開判断 → デスク（Managing Editor）
- 画像の作成・配置 → UI/フロント担当
- `heroScope` / `pinned` の設定 → デスク

---

## 出力物

1. **推敲済み記事オブジェクト**
2. **編集メモ**:
   - 変更した箇所と理由
   - 記者への差し戻し項目（あれば）
   - UI担当への依頼事項（図表・画像が必要な場合）

## まとめ記事の構成ガイド

速報ニュースが蓄積されたテーマについて、**体系的なまとめ記事**を設計する。

### まとめ記事の種類

| 種類 | 束ね方 | 例 |
|------|--------|-----|
| **製品別まとめ** | 1つの製品の機能・更新を時系列で整理 | 「Claude Code 機能一覧と進化」 |
| **企業別まとめ** | 1社の動向を俯瞰 | 「Anthropic の2026年 — モデル・法廷・製品」 |
| **テーマ別まとめ** | 複数社にまたがるトピックを横断整理 | 「生成音楽ツールの現在地」 |
| **週刊まとめ** | 1週間の出来事を時系列で俯瞰 | `weekly-news-roundup` スキルに従う |

### まとめ記事の構成テンプレート

```
title: 「【まとめ】{製品名/テーマ} — {時期または副題}」
type: "feature"
heroScope: "none"（通常）または "day"（特に重要な場合）

body の構成:
1. リード — このまとめが扱う範囲と、最も重要な変化
2. 時系列 or テーマ別ブロック — 各速報記事の要点を統合
3. 各ブロック末尾に ?a=速報記事ID で深掘りリンク
4. 締め — 現時点での全体像（感想ではなく事実の要約）
```

### ニュース → まとめ の判断基準

- 同一製品/テーマの速報が **3本以上** → まとめ記事を検討
- 記者から提案があった場合 → 骨格を設計して返す
- 既存のまとめ記事がある場合 → 更新（新規作成ではなく追記）を優先

- [ ] リードが第1段落にあるか

- [ ] 同じ接続詞が3回以上連続していないか
- [ ] 同じ日の出来事が1段落に束ねられているか
- [ ] 段落間の関係（時系列/因果/対比）が明示されているか
- [ ] `weekly-news-roundup` スキルの品質チェックに合格するか（週刊記事の場合）
