# 記者（Reporter） — 取材・下書きスキル

## 役割の定義

編集部における**一次情報の収集者・下書き執筆者**。公式発表・報道・技術ブログから事実を拾い、記事の初稿を作成する。

---

## 取材フロー

### Step 1 — ネタの発見と評価

1. **ソースの優先順位**（信頼度順）:
   - 公式プレスリリース・ブログ（企業サイト、GitHub リリースノート）
   - 主要通信社・新聞（Reuters, AP, NYT, Bloomberg）
   - 技術メディア（The Verge, TechCrunch, Ars Technica）
   - 開発者コミュニティ（Hacker News, X/Twitter の公式アカウント）
2. **掲載判断の基準**:
   - AI 開発ツール・モデル・企業に関連するか
   - 読者（日本語圏の開発者）にとって実用的か
   - 事実として確認可能か（「噂」「リーク」は明示する）

3. **取材カテゴリ**（網羅すべき領域）:

   | カテゴリ | 具体例 |
   |---------|--------|
   | バイブコーディング | Claude Code, Cursor, Windsurf, Copilot, Codex の新機能 |
   | AI モデル | GPT, Claude, Gemini, Llama の新バージョン・ベンチマーク |
   | 画像生成 | Midjourney, DALL-E, Firefly, Stable Diffusion |
   | 動画生成 | Runway, Sora, Veo, Kling |
   | 音楽生成 | Suno, Udio, Lyria |
   | 音声入力・TTS | Aqua Voice, Whisper, ElevenLabs |
   | フィジカル AI | ロボティクス, 自動運転, エッジ AI |
   | 社会・規制 | AI 規制, 訴訟, 企業動向, 資金調達 |
   | 実用スキル | スキルファイルの書き方, CLAUDE.md, RAG, 音声コマンド活用 |

### Step 2 — ファクトの構造化

収集した事実を以下の形で整理する:

| 項目 | 記入内容 |
|------|---------|
| **何が起きたか** | 主語 + 動詞 + 目的語（一文） |
| **いつ** | 日付（YYYY-MM-DD、東京時間基準） |
| **誰が** | 企業名・人名（`aiCompanies.js` の `id` と紐付け可能か確認） |
| **ソース** | URL + サイト名（`primarySources` 形式で記録） |
| **関連する既存記事** | `?a=article-id` があれば記録 |

### Step 3 — 下書き執筆

**記事データの形式**（`aiToolsData.js` に準拠）:

```javascript
{
  id: "kebab-case-slug",        // 一意のスラッグ
  type: "news",                  // news | feature | review
  category: "model",             // model | cli | editor | product | ...
  title: "見出し — 簡潔に事実を",
  excerpt: "概要。**太字**で主語を強調。事実のみ、感想禁止。",
  body: [
    "第1段落 — リード。最も重要な事実。",
    "第2段落 — 補足情報・背景。",
    "第3段落 — 関連リンク ?a=... への導線。"
  ],
  date: "YYYY-MM-DD",
  author: "AI News 編集部",
  readTime: "N分",
  tags: ["関連タグ"],
  primarySources: [
    { title: "ソースタイトル", site: "サイト名", url: "https://..." }
  ]
}
```

## 記者の禁止事項

- 意見・感想を書かない（「素晴らしい」「期待される」「注目だ」は構成編集の判断）
- 未確認情報を断定しない（「と報じられた」「と伝えられた」を使う）
- 他記事のコピペ禁止 — 必ず自分の言葉で再構成する
- heroScope・pinned の設定はしない — デスク（Managing Editor）の管轄

## テキスト装飾ルール

- `**太字**` は最小限。固有名詞の初出程度。1段落に3箇所以上は多すぎ
- `==ハイライト==` は読者に注目してほしい箇所（背景色が付く）
- `!!警告!!` は危険操作・絶対にやってはいけないこと（赤文字になる）
- 本文の大部分は通常テキストで書く。フラットな文章が読みやすい

---

## 出力物

記者の成果物は以下の2つ:

1. **記事オブジェクトの下書き**（上記フォーマット）
2. **取材メモ**（ソース一覧、確認済み/未確認の区分、関連記事ID）

下書きは**校閲（Copy Editor）→ 構成編集（Editor）→ デスク（Managing Editor）**の順でレビューされる。

---

## ニュース → まとめ記事への橋渡し

速報記事を書くとき、以下を意識する:

1. **タグの一貫性**: まとめ記事で束ねやすいよう、製品名・企業名タグを正確に付ける
2. **まとめ候補の提案**: 同じ製品・テーマの速報が3本以上溜まったら、構成編集に「まとめ記事の起稿」を提案する
3. **内部リンクの設計**: 速報記事の `body` に `?a=まとめ記事ID` を入れられるよう、将来のまとめ記事IDを想定しておく

**まとめ記事の起稿例**:
```
速報3本以上 → 記者が構成編集に提案 → 構成編集がまとめ記事の骨格を設計 → 記者が情報を統合して下書き
```
