# データ品質 — メタデータ整合性と出稿判断

## 1. 企業プロファイル管理（aiCompanies.js）

### データ構造

```javascript
{
  id: "kebab-case",              // 一意の識別子
  name: "表示名",
  legalName: "Legal Name Inc.",
  country: "アメリカ合衆国",
  hq: "カリフォルニア州サンフランシスコ",
  foundedYear: 2021,
  employees: "数百〜1,000名規模（報道・推計）",
  revenue: "非公開（推定ARRの報道あり）",
  stock: {
    listed: true | false,
    tickers: [{ symbol: "TICK", exchange: "NASDAQ", name: "普通株" }],
    detail: "補足説明"
  },
  products: ["製品A", "製品B"],
  officialUrl: "https://...",
  notes: ["補足事項"]
}
```

### 企業情報を載せる理由（読者価値）

| 読者の疑問 | 企業情報がどう役立つか |
|-----------|----------------------|
| このツール、どこの会社？ | `products` と `name` の紐付け |
| 逆に〇〇社って何を出してる？ | `products` 一覧 |
| これって中国？アメリカ？ | `country` / `hq` |
| 株買えるの？上場してるの？ | `stock.tickers` / `stock.listed` |

### 更新ルール

| 項目 | 更新頻度 | ソース |
|------|---------|--------|
| products | 記事で新製品が登場するたび | 公式発表 |
| employees | 四半期〜半年ごと | 有価証券報告書・報道 |
| revenue | 年次 | 決算報告・報道 |
| stock | IPO・上場廃止・資金調達時 | 取引所発表・報道 |

### 新規企業の追加基準

- AI-News の記事に2回以上登場する企業
- AI 開発ツール・モデルに直接関連する企業
- `_AI_COMPANIES_UNSORTED` 配列に追加（ソートは自動）
- 新規レビュー記事作成時は企業データも必ずセットで確認・追加

### 株式情報の管理

- 上場企業: `stock.listed: true`、`tickers` に証券コード・取引所を記載
- 未上場企業: `stock.listed: false`、`detail` に出資元・資金調達の概要
- 投資判断を促す表現は禁止（「注目」「買い時」等は書かない）

## 2. タグ体系・カテゴリ定義

### カテゴリ一覧

```
special    — 週刊まとめ・特集
model      — AIモデル（GPT, Claude, Gemini 等）
cli        — CLIツール（Claude Code, uv 等）
editor     — エディタ・IDE（Cursor, Windsurf 等）
data       — データ・分析ツール
product    — 製品・サービス全般
media      — メディア生成（画像・動画・音声）
regulation — 社会・規制・セキュリティ
```

### タグの命名規則

- 日本語を基本とする（「エージェント」「コード生成」）
- 英語のままが自然なもの（「MCP」「RAG」「Claude」）はそのまま
- 年月タグ: 「2026年3月」形式
- 新タグ追加時は既存タグとの重複・包含関係を確認

### 正規化ルール

- 同一概念のタグが複数存在しないこと（「エージェント」と「Agent」の混在禁止）
- `aiCompanies.js` の `name` と記事内の企業名表記を一致させる
- `type` は `news | feature | review` のいずれか

## 3. 用語集（vibeCodingGuide.js）の更新ルール

- `VIBE_PROGRESSION_PATH` — 段階的な道筋（更新時は全段階の整合性を確認）
- `VIBE_TOOL_COMBO_TABLE` — 組み合わせ表（ツールの追加・廃止を反映）
- `VIBE_CODING_WHY_CLAUDE_FIRST` — まず Claude を勧める理由（公式料金改定時に更新）
- ガイドタブ = 概要レベル、記事タブ = 詳細な時点付き評価。矛盾させない

## 4. メタデータ整合性の監査基準

定期的に以下を検証する:

1. ARTICLES の全 `id` が一意か
2. `body` 内の `?a=xxx` が実在する記事を指しているか
3. `tags` の表記揺れがないか
4. `category` が定義済みの値か
5. `aiCompanies.js` の企業が記事で参照されているか（孤立データの検出）
6. `date` が妥当な範囲か（未来日付の意図的な設定を除く）
7. `readTime` が本文量と整合しているか（日本語 400-600文字/分）

## 5. 出稿判断基準

### 出稿可否

以下がすべて OK なら出稿許可:

| 条件 | 確認方法 |
|------|---------|
| 校閲完了 | 未解決の事実誤認・リンク切れがないか |
| 構成編集完了 | 差し戻し項目がないか |
| ソース確保 | `primarySources` が1つ以上、URL が有効 |
| ビルド通過 | `npm run build` がエラーなし |
| 重複なし | 同じネタの既存記事がないか |

### 差し戻し基準

- ソースが1つもない → 取材に差し戻し
- 事実の不整合 → ファクトチェックに差し戻し
- 構成が読みにくい → 推敲に差し戻し

## 6. heroScope・pinned・date の決定ルール

### heroScope

| 値 | 用途 | 条件 |
|----|------|------|
| `"day"` | 当日のヒーロー記事 | その日の最重要ニュース |
| `"week"` | 週刊まとめのヒーロー | `weekRoundupPeriod` 必須。毎週月曜公開 |
| `"none"` | ヒーローに出さない | 補足記事・過去記事の更新 |

### pinned

- `pinned: true` — 同日に複数記事がある場合にヒーロー候補の優先権を与える
- 原則として1日1記事のみ pinned
- 週刊まとめは自動的に `pinned: true`

### 公開日（date）の決定

- 速報: ニュース発生日（東京時間）
- 週刊まとめ: 対象週の翌月曜
- レビュー・特集: 執筆完了後の最適日（他記事との重複を避ける）

### 最終確定フィールド

```javascript
{
  heroScope: "day" | "week" | "none",
  pinned: true | false,
  date: "YYYY-MM-DD",
  newsDate: "YYYY-MM-DD",             // 必要に応じて
  weekRoundupPeriod: { start, end }   // 週刊まとめのみ
}
```

## 7. 配信スケジュール

### 週間カレンダー

```
月曜: 週刊まとめ公開（前週分）
火〜金: 速報・レビュー（1日最大2本が目安）
土日: 特集・ガイド更新（緊急でなければ避ける）
```

同日公開の上限: 原則2本。3本以上になる場合は翌日にずらす。

### 1週間の理想的な構成

- 速報/短報: 5〜10本
- まとめ更新: 1〜2本
- 週刊まとめ: 1本（月曜公開）
- ガイド更新: 必要に応じて

## 8. 記事間依存関係の管理

### 速報とまとめの相互リンク

- 速報 → まとめ: 速報記事の `body` にまとめ記事への `?a=` リンクを追加
- まとめ → 速報: まとめ記事の各ブロック末尾に元の速報への `?a=` リンク
- 新しい速報が出たら、関連するまとめ記事にもリンクを追加

### まとめ記事の発注判断

| 条件 | アクション |
|------|-----------|
| 同一テーマの速報が3本以上 | まとめ記事の設計を開始 |
| 既存まとめが古くなった | 更新版を制作（新規IDではなく既存記事を更新） |

### ARTICLES 配列の挿入位置

新記事は配列の先頭（最新が上）に挿入する。
