# 校閲（Copy Editor） — 事実確認・品質保証スキル

## 役割の定義

記者が書いた下書きの**正確性・一貫性・リンクの有効性**を検証する。文体や構成には踏み込まず、「事実として正しいか」「表記が統一されているか」に集中する。

---

## 検証フロー

### Step 1 — ファクトチェック

| チェック項目 | 方法 |
|-------------|------|
| **日付の正確性** | `primarySources` の URL を開き、公式発表日と記事の `date` / `newsDate` が一致するか |
| **企業名・製品名** | `aiCompanies.js` の `name` / `products` と照合。正式名称を使っているか |
| **数値・バージョン** | 公式ソースの数値と記事内の記述が一致するか |
| **人名** | 公式プロフィール・LinkedIn 等で確認（肩書きが古くないか） |
| **引用の正確性** | 「と述べた」「と発表した」の内容が元ソースと合致するか |

### Step 1.5 — チャート・グラフデータの検証（重要）

記事に `charts` フィールドがある場合、全数値の出典を確認する。

| チェック項目 | 方法 |
|-------------|------|
| 数値の出典 | 各 `bars[].value` が `primarySources` のソースに記載されているか |
| 推定値の明示 | ソースで確認できない値は subtitle に「推定値を含む」と明記 |
| 捏造の禁止 | ==ソースがない数値でチャートを作らない。推測で「だいたいこのくらい」の値を入れるのは禁止== |
| 単位の正確性 | `unit` フィールドが正しいか（%, x, 万 等） |
| maxValue の妥当性 | 最大値がデータを見やすく表現しているか |

!!ソースのないデータでグラフを作ると、サイトの信頼性が根本から崩れる。推定値を使う場合は必ず「推定」とラベルする。!!

### Step 2 — リンク検証

```
検証対象:
1. primarySources[].url — 全リンクがアクセス可能か
2. body 内の ?a=article-id — 対応する記事が ARTICLES 配列に存在するか
3. body 内の https:// リンク — アクセス可能か（404/403 でないか）
```

**検証コマンド（内部リンク）**:
- `ARTICLES` 配列内で `id` を grep し、`?a=` で参照されている ID が実在するか確認

### Step 3 — 表記揺れ修正

**統一ルール**:

| 項目 | 正 | 誤（よくある揺れ） |
|------|----|-------------------|
| 企業名 | `aiCompanies.js` の `name` | 略称・旧名の混在 |
| 製品名 | 公式表記（大文字小文字含む） | Claude code → Claude Code |
| 日付形式 | YYYY-MM-DD（データ）/ M月D日（本文） | 2026/03/28, 3/28 |
| 装飾記法 | `**太字**` `==ハイライト==` `!!警告!!` | `*イタリック*`（未対応） |
| 敬称 | 原則なし（報道文体） | 〜氏、〜さん（統一されていれば可） |
| 句読点 | 「、」「。」 | 「，」「．」混在 |

太字の使いすぎチェック（校閲の重要な確認事項）:
- 1段落に `**` が3箇所以上 → 構成編集に「太字減らし」を提案
- 固有名詞の初出以外に太字が使われていたら指摘
- 注意喚起は `==ハイライト==`、禁止事項は `!!警告!!` を推奨

### Step 4 — メタデータ検証

```
確認項目:
- [ ] id: kebab-case で一意か
- [ ] type: news | feature | review のいずれか
- [ ] category: 定義済みカテゴリか（special, model, cli, editor, data, product, voice, vision, video）
- [ ] tags: 既存タグとの表記揺れがないか
- [ ] readTime: 本文量に対して妥当か（日本語 400-600文字/分 で算出）
- [ ] author: "AI News 編集部" が標準
```

---

## 校閲の権限と制限

**できること**:
- 誤字・脱字の修正
- 事実誤認の指摘と修正提案
- リンク切れの報告
- 表記揺れの統一

**できないこと（他ロールの管轄）**:
- 記事の構成変更 → 構成編集（Editor）
- 見出しの書き換え → 構成編集（Editor）
- 公開判断・heroScope 設定 → デスク（Managing Editor）
- 画像・図表の追加 → UI/フロント担当

---

## 出力物

校閲の成果物:

1. **修正済み記事オブジェクト**（変更箇所にコメント付き）
2. **校閲レポート**:
   - 修正した項目のリスト
   - 未解決の疑問点（記者に差し戻す項目）
   - リンク検証結果（OK / NG / 要確認）
